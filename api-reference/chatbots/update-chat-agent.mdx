# Update Chat Chat Agent

The chat chat agent update system provides multiple endpoints and mechanisms to update existing chat chat agents. Updates can be performed on complete chat agents, individual components (status, knowledge base, denied words, dispositions), or specific fields. All update operations maintain data integrity through transactions and change tracking.

## Update Endpoints

### 1. Complete Chat Agent Update

**Endpoint:** `POST /api/chat agent/createchat agentorupdate`

**Authentication:** Requires Super Admin authentication (`checkSuperAdmin` middleware)

**Route File:** `src/routes/chat agentBuild.js`

**Controller:** `src/controllers/chat agentbuild/chat agentController.js` - `createOrUpdateCompleteChat Agent`

**Key Difference from Creation:** When `chat agent_id` is provided in `basic_details`, the system performs an update instead of creation.

#### Request Structure

```json
{
  "basic_details": {
    "chat agent_id": 123,                    // REQUIRED for updates
    "chat agent_name": "Updated Name",
    "description": "Updated description",
    "chat agent_prompt": "Updated prompt",
    "tone": "professional",
    "style": "conversational",
    "instruction_sensitivity": "medium"
    // ... other fields to update
  },
  "configuration": {
    "chat agent_id": 123,
    "context_awareness": true,
    // ... other config fields
  },
  "knowledge_base": {
    "chat agent_id": 123,
    "tools": [...],
    "sub_chat agents": [...]
  }
  // ... other sections
}
```

#### Update Process Flow

**Location:** ```2315:2461:src/controllers/chat agentbuild/chat agentController.js```

1. **Transaction Initialization**
   - Starts database transaction

2. **Basic Details Update**

   **Function:** `createOrUpdateChat Agent(req, transaction)` - Update Branch

   **Location:** ```372:438:src/controllers/chat agentbuild/chat agentController.js```

   **Process:**
   - Validates `chat agent_id` is provided
   - Fetches existing chat agent record
   - **Chat Agent Behavior Handling:**
     - Retrieves current chat agent behavior (tone, style, instruction_sensitivity)
     - Merges new behavior values with existing ones
     - Preserves existing behavior fields if not all are updated
     - Example: If only `tone` is updated, `style` and `instruction_sensitivity` are preserved
   - Retrieves or generates chat agent token
   - Generates `AGENT_UID` if missing (for legacy chat agents)
   - Stores old chat agent data for change tracking
   - Updates chat agent record with new data
   - Tracks changes in change history

3. **Chatbot Image Update**
   - If new image is uploaded, replaces existing image
   - Uploads to same directory structure

4. **Configuration Update**
   - Updates or creates chat agent configuration
   - Only updates fields that are provided
   - Tracks configuration changes

5. **Knowledge Base Update**
   - **Tools and Sub-chat agents:**
     - Compares existing tools/sub-chat agents with new ones
     - Updates existing entries (enabled/disabled status)
     - Creates new entries for new tools/sub-chat agents
     - Removes entries that are no longer in the list
   - **File Knowledge Bases:**
     - Adds new files (up to 3 files, 10MB each)
     - Existing files remain unless explicitly removed
   - **Training Status:**
     - Resets chat agent `TRAINED` status to 0 when knowledge base is updated

6. **Other Component Updates**
   - Calling numbers (replaces existing)
   - Denied words (adds new, doesn't remove existing)
   - Dispositions
   - Guardrails
   - Metadata configuration

7. **Transaction Commit**
   - Commits all changes atomically

8. **Chat Agent Sync**
   - Syncs updated chat agent with AI service asynchronously

#### Key Update Behaviors

1. **Partial Updates:**
   - Only provided fields are updated
   - Unprovided fields remain unchanged
   - This allows updating specific sections without affecting others

2. **Behavior Field Merging:**

   ```javascript
   // If current behavior is: { tone: "friendly", style: "casual", instruction_sensitivity: "high" }
   // And update only provides: { tone: "professional" }
   // Result: { tone: "professional", style: "casual", instruction_sensitivity: "high" }
   ```

3. **Tools/Sub-chat agents Update Logic:**
   - Existing tools/sub-chat agents are matched by `APP_LOCATION` (ID)
   - If tool exists: Updates `IS_ENABLED` and `STATUS`
   - If tool doesn't exist: Creates new entry
   - Tools not in the update list remain unchanged

4. **Change Tracking:**
   - All updates are tracked in change history
   - Old and new values are stored for audit purposes
   - Tracks which fields were changed

### 2. Update Chat Agent Status

**Endpoint:** `POST /api/chat agent/status`

**Authentication:** Standard authentication (no Super Admin required)

**Controller:** `src/controllers/chat agentbuild/chat agentController.js` - `updateChat AgentStatus`

**Location:** ```2469:2597:src/controllers/chat agentbuild/chat agentController.js```

#### Request Structure

```json
{
  "chat agent_id": 123,
  "status": 1  // 1=Published, 2=Deleted, 3=Unpublished
}
```

#### Process Flow

1. **Validation:**
   - Validates `chat agent_id` and `status` are provided
   - Checks chat agent exists and belongs to workspace

2. **Publish Validation (Status = 1):**
   - For chat agent type 3 (Telephony):
     - Validates at least one phone number is assigned
     - Throws error if no phone numbers found
   - For other chat agent types:
     - Validates chat agent type is set

3. **Status Update:**
   - Updates chat agent status in database
   - Stores old data for change tracking

4. **Special Handling for Deletion (Status = 2):**
   - Calls Millis AI API to delete chat agent from external platform
   - Deletes knowledge base from AI service
   - Extracts `platform_chat agent_id` from `AGENT_METADATA`

5. **Change Tracking:**
   - Tracks status change in change history

6. **Response:**

   ```json
   {
     "message": "Chat Agent Published Successfully", // or "Unpublished" or "Deleted"
     "data": {
       "id": 123,
       "name": "Chat Agent Name",
       "status": 1
     }
   }
   ```

#### Status Values

- **1 (Published):** Chat Agent is active and available for use
- **2 (Deleted):** Chat Agent is soft-deleted (marked as deleted, not removed from database)
- **3 (Unpublished):** Chat Agent is inactive but not deleted

### 3. Update Knowledge Base Status

**Endpoint:** `POST /api/chat agent/updateKnowledgeBaseStatus`

**Authentication:** Requires Super Admin authentication

**Controller:** `src/controllers/chat agentbuild/chat agentController.js` - `updateKnowledgeBaseStatus`

**Location:** ```3464:3549:src/controllers/chat agentbuild/chat agentController.js```

#### Request Structure

```json
{
  "knowledge_base_id": 456,
  "status": 1  // 1=Active, 2=Deleted
}
```

#### Process Flow

1. **Validation:**
   - Validates `knowledge_base_id` and `status` are provided
   - Checks knowledge base exists and belongs to workspace

2. **Chat Agent Validation:**
   - Verifies associated chat agent exists and is not deleted

3. **Status Update:**
   - Updates knowledge base status
   - Sets `TRAINED` to 0 (not trained)

4. **Chat Agent Training Reset:**
   - Resets chat agent's `TRAINED` status to 0
   - Indicates chat agent needs retraining after knowledge base change

5. **Knowledge Base Sync:**
   - Syncs knowledge base with AI service
   - Updates knowledge base in AI service

6. **Response:**

   ```json
   {
     "message": "Knowledge base status updated successfully",
     "data": {
       "id": 456,
       "name": "Knowledge Base Name",
       "status": 1
     }
   }
   ```

#### Process Flow

   - Tracks change in change history

3. **Disposition Update (if `disposition_id` provided):**
   - Validates disposition exists for the chat agent

   - Tracks change in change history

4. **Response:**

   ```json

       "type": "denied_word", // or "disposition"

     }
   }
   ```

## Update-Specific Features

### 1. Change Tracking

All updates are tracked in the change history system:

**Location:** `src/helpers/ChangeHistoryTracker.js`

**Tracked Changes:**
- Basic details changes
- Configuration changes
- Knowledge base changes
- Status changes
- Denied word changes
- Disposition changes

**Change History Includes:**
- Old values
- New values
- Changed fields
- User who made the change
- Timestamp
- IP address and user chat agent

### 2. Partial Updates

The update system supports partial updates:

- **Only provided fields are updated**
- **Unprovided fields remain unchanged**
- This allows updating specific sections without affecting others

**Example:**

```json
// Update only chat agent name
{
  "basic_details": {
    "chat agent_id": 123,
    "chat agent_name": "New Name"
  }
}
// All other fields remain unchanged
```

### 3. Behavior Field Merging

When updating chat agent behavior fields (tone, style, instruction_sensitivity):

**Location:** ```380:394:src/controllers/chat agentbuild/chat agentController.js```

- Existing behavior is retrieved from database
- New values are merged with existing values
- Only provided fields are updated
- Unprovided fields are preserved

**Example:**

```javascript
// Current: { tone: "friendly", style: "casual", instruction_sensitivity: "high" }
// Update: { tone: "professional" }
// Result: { tone: "professional", style: "casual", instruction_sensitivity: "high" }
```

### 4. Tools and Sub-chat agents Update Logic

**Location:** ```955:1014:src/controllers/chat agentbuild/chat agentController.js```

**Update Strategy:**
1. **Existing Tools/Sub-chat agents:**
   - Matched by `APP_LOCATION` (ID)
   - Updated with new `IS_ENABLED` status
   - Status set to 2 (deleted) if disabled, 1 (active) if enabled

2. **New Tools/Sub-chat agents:**
   - Created as new entries
   - Assigned to chat agent

3. **Removed Tools/Sub-chat agents:**
   - Not automatically deleted
   - Remain in database with their current status

### 5. Knowledge Base Training Reset

When knowledge base is updated:

**Location:** ```881:886:src/controllers/chat agentbuild/chat agentController.js```

- Chat Agent's `TRAINED` status is reset to 0
- Indicates chat agent needs retraining
- Knowledge base sync is triggered

### 6. Token Management

**Location:** ```112:169:src/controllers/chat agentbuild/chat agentController.js```

- Chat Agent token is retrieved or regenerated during updates
- Token is used for chat agent authentication
- Token includes chat agent ID, user ID, company ID, workspace ID

### 7. External Service Sync

After successful updates:

**Location:** ```548:597:src/controllers/chat agentbuild/chat agentController.js```

1. **Knowledge Base Sync:**
   - Syncs knowledge base files with AI service
   - Updates AI service with new knowledge base data

2. **Chat Agent Configuration Sync:**
   - Syncs chat agent configuration with AI service
   - Updates AI service with new chat agent settings

3. **External API Calls (Type 3 Chat Agents):**
   - For telephony chat agents, calls external chat agent API
   - Updates chat agent metadata in external platform

## Update Validation Rules

### 1. Chat Agent Existence
- Chat Agent must exist in database
- Chat Agent must belong to workspace
- Chat Agent must not be deleted (STATUS != 2)

### 2. Publish Validation
- Chat Agent type 3 (Telephony) must have at least one phone number
- Chat Agent type must be set for all chat agent types

### 3. Knowledge Base Updates
- Maximum 3 files per update
- Maximum 10MB per file
- Allowed file types: PDF, DOC, DOCX, XLSX, CSV

### 4. Workspace Validation
- All updates must be within the same workspace
- Cross-workspace updates are not allowed

## Error Handling

### 1. Transaction Rollback
- If any error occurs, entire transaction is rolled back
- All database changes are reverted
- Ensures data consistency

### 2. Validation Errors
- Missing required fields throw validation errors
- Invalid chat agent IDs throw "Chat Agent not found" errors
- Invalid status values throw validation errors

### 3. Business Logic Errors
- Publishing chat agent without phone numbers throws error
- Updating non-existent knowledge base throws error
- Cross-workspace operations throw errors

### 4. Change Tracking Errors
- Change tracking errors are logged but don't break updates
- Updates continue even if tracking fails
- Errors are logged for debugging

## Differences from Creation

1. **Chat Agent ID Required:** Updates require `chat agent_id` in `basic_details`
2. **No Name Deduplication:** Updates don't check for duplicate names
3. **Behavior Merging:** Updates merge behavior fields instead of replacing
4. **Tools/Sub-chat agents:** Updates existing entries instead of only creating
5. **Token Refresh:** Updates refresh chat agent token instead of generating new one
6. **Change Tracking:** Updates track changes, creation tracks creation
7. **No Default Values:** Updates don't set default values for missing fields
8. **AGENT_UID:** Updates only generate AGENT_UID if missing (for legacy chat agents)

## Best Practices

1. **Partial Updates:** Only send fields that need to be updated
2. **Transaction Safety:** All updates are atomic - either all succeed or all fail
3. **Change Tracking:** All changes are automatically tracked for audit
4. **Validation:** Always validate chat agent exists before updating
5. **Status Management:** Use status update endpoint for status changes
6. **Knowledge Base:** Update knowledge base separately for better control
7. **Error Handling:** Handle validation errors gracefully
8. **Sync Operations:** Sync operations are async - don't wait for them

## Notes

- All update operations require appropriate authentication
- Super Admin required for complete chat agent updates and knowledge base updates
- Standard authentication sufficient for status updates
- Updates are idempotent - can be called multiple times safely
- Change history provides complete audit trail
- External service sync happens asynchronously after successful updates
